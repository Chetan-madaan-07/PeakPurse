# Requirements Document

## Introduction

PeakPurse is a web-based intelligent personal finance platform targeted at Indian users. It unifies budgeting, tax filing, CA discovery, social benchmarking, subscription tracking, investment planning, and a conversational AI assistant into a single integrated experience. The platform's primary goal is to improve users' financial health through automation, explainable insights, and goal-driven recommendations — all while complying with Indian regulations including the DPDP Act 2023, Income Tax Act, SEBI RIA regime, and RBI digital guidelines.

The platform is delivered initially as a responsive web application (mobile-first) and is structured around seven core feature modules, a shared authentication and consent layer, and a set of cross-cutting non-functional concerns including security, performance, observability, and regulatory compliance.

## Glossary

- **Platform**: The PeakPurse intelligent personal finance web application.
- **User**: An authenticated individual using the Platform to manage personal finances.
- **Admin**: A privileged operator of the Platform with access to the admin dashboard and configuration tools.
- **CA**: A Chartered Accountant listed in the Platform's CA Directory.
- **Auth_Service**: The service responsible for user registration, login, token management, and consent management.
- **Finance_Service**: The service responsible for accounts, transactions, categories, budgets, goals, subscriptions, and financial health scores.
- **Tax_Service**: The service responsible for tax profiles, deduction modeling, tax-form data structures, and export of ITR-ready data.
- **CA_Service**: The service responsible for CA profiles, discovery, filtering, and lead routing.
- **Benchmark_Service**: The service responsible for cohort definitions, anonymized aggregates, and percentile calculations.
- **Chatbot_Service**: The service responsible for intent detection, dialog management, and orchestration to other services.
- **Investment_Service**: The service responsible for risk profiling, goal-based plan generation, and recommended allocations.
- **Notification_Service**: The service responsible for email, push, and in-app notifications and reminders.
- **Audit_Service**: The service responsible for audit trails, data access logs, and compliance logs.
- **ML_Service**: The Python-based service responsible for financial health scoring models and recommendation engine.
- **Categorizer**: The component within Finance_Service that classifies transactions into categories using rule-based and ML methods.
- **Health_Score**: A 0–100 composite score representing a User's financial health, computed by the ML_Service.
- **Cohort**: A group of anonymized Users sharing similar income range, age band, city tier, family status, and risk profile, used for benchmarking.
- **Subscription**: A detected recurring charge in a User's transaction history, tracked by the Finance_Service.
- **TaxProfile**: A per-financial-year record of a User's income, deductions, regime selection, and filing status.
- **InvestmentPlan**: A goal-linked set of recommended asset category allocations and SIP amounts generated by the Investment_Service.
- **ConsentRecord**: A timestamped record of a User's explicit consent for a specific data processing purpose, managed by the Auth_Service.
- **ERI**: An e-Return Intermediary registered with the Income Tax Department of India, authorized to file ITRs on behalf of taxpayers.
- **RIA**: A SEBI-registered Investment Adviser authorized to provide individualized investment advice for a fee.
- **DPDP Act**: The Digital Personal Data Protection Act 2023, governing collection and processing of digital personal data in India.
- **FY**: Financial Year (April 1 to March 31) as defined under Indian tax law.
- **80C / 80D / 80CCD**: Sections of the Indian Income Tax Act providing deductions for specified investments and insurance premiums.
- **SIP**: Systematic Investment Plan — a method of investing a fixed amount periodically in mutual funds.
- **ELSS**: Equity Linked Savings Scheme — a tax-saving mutual fund category eligible for 80C deduction.
- **NPS**: National Pension System — eligible for 80CCD deduction.
- **AA**: Account Aggregator — an RBI-regulated entity enabling consent-based financial data sharing.
- **ITR**: Income Tax Return — the form filed with the Indian Income Tax Department.
- **TDS**: Tax Deducted at Source — tax withheld by the payer before crediting income to the recipient.
- **Form 26AS**: A consolidated annual tax statement issued by the Income Tax Department showing TDS, TCS, and advance tax details.

## Requirements

### Requirement 1: User Registration and Authentication

**User Story:** As a new visitor, I want to register and securely log in to the Platform, so that my financial data is protected and accessible only to me.

#### Acceptance Criteria

1. THE Auth_Service SHALL accept a registration request containing a unique email address, a password meeting minimum complexity rules (at least 8 characters, at least one uppercase letter, at least one digit, and at least one special character), and an explicit acceptance of the Terms of Service and Privacy Policy.
2. WHEN a registration request is received with a duplicate email address, THE Auth_Service SHALL return an error response with a machine-readable code indicating the email is already registered.
3. WHEN a User submits valid registration credentials, THE Auth_Service SHALL create a User record, send an email verification link to the provided address, and return a success response within 2 seconds.
4. WHEN a User clicks a valid email verification link, THE Auth_Service SHALL mark the User's email as verified and allow login.
5. WHEN a User submits valid login credentials, THE Auth_Service SHALL issue a signed JWT access token with a 15-minute expiry and a refresh token with a 7-day expiry.
6. WHEN a login attempt is made with an unverified email address, THE Auth_Service SHALL return an error response indicating email verification is required.
7. WHEN 5 consecutive failed login attempts occur for the same account within a 10-minute window, THE Auth_Service SHALL temporarily lock the account for 15 minutes and notify the User via email.
8. THE Auth_Service SHALL support time-based one-time password (TOTP) two-factor authentication as an optional security enhancement for any User account.
9. WHEN a User enables 2FA, THE Auth_Service SHALL require a valid TOTP code on every subsequent login before issuing tokens.
10. WHEN a refresh token is presented before its expiry, THE Auth_Service SHALL issue a new access token and rotate the refresh token.
11. WHEN a User requests logout, THE Auth_Service SHALL invalidate the current refresh token within 500 milliseconds.
12. THE Auth_Service SHALL enforce role-based access control with at minimum three roles: User, CA, and Admin, and SHALL reject requests to resources outside a caller's role permissions with an HTTP 403 response.

### Requirement 2: Consent Management

**User Story:** As a User, I want granular control over how my data is used and shared, so that I can trust the Platform and comply with my rights under the DPDP Act 2023.

#### Acceptance Criteria

1. THE Auth_Service SHALL present a distinct, purpose-specific consent prompt before collecting or processing data for each of the following purposes: transaction categorization, benchmarking aggregation, CA profile sharing, and marketing communications.
2. WHEN a User grants consent for a specific purpose, THE Auth_Service SHALL create a ConsentRecord containing the purpose identifier, timestamp, and User identifier, and SHALL store it in durable storage.
3. WHEN a User revokes consent for a specific purpose, THE Auth_Service SHALL update the corresponding ConsentRecord with a revocation timestamp within 1 second and SHALL cease processing data for that purpose within 24 hours.
4. THE Auth_Service SHALL provide an endpoint that returns all active ConsentRecords for the authenticated User.
5. WHEN a User submits a data deletion request, THE Auth_Service SHALL initiate deletion of all personal data associated with that User across all services within 30 days, subject to legally mandated retention obligations.
6. IF a data processing operation is attempted for a purpose for which the User has not granted consent or has revoked consent, THEN THE Auth_Service SHALL reject the operation and log the rejection in the Audit_Service.
7. THE Audit_Service SHALL retain consent grant and revocation events for a minimum of 3 years to support regulatory audit requirements.

### Requirement 3: Account and Transaction Management

**User Story:** As a User, I want to connect my bank accounts and import transactions, so that the Platform can automatically analyze my income and expenses.

#### Acceptance Criteria

1. THE Finance_Service SHALL allow a User to create manual Account records specifying account type (bank, credit card, wallet, UPI, or other), provider name, and currency (defaulting to INR).
2. THE Finance_Service SHALL accept bank statement uploads in PDF and CSV formats and parse them into Transaction records associated with the corresponding Account.
3. WHEN a statement file is uploaded, THE Finance_Service SHALL complete parsing and return a summary of imported transactions within 60 seconds for files up to 10 MB.
4. IF a statement file cannot be parsed due to an unrecognized format, THEN THE Finance_Service SHALL return a descriptive error message identifying the unsupported format and SHALL not create partial Transaction records.
5. THE Finance_Service SHALL support integration with Account Aggregator (AA) networks to fetch transaction data with User consent, subject to AA availability and regulatory permissions.
6. WHEN AA-sourced transaction data is fetched, THE Finance_Service SHALL verify that a valid ConsentRecord exists for the transaction categorization purpose before storing the data.
7. THE Finance_Service SHALL allow a User to manually create, edit, and delete individual Transaction records including date, amount, merchant name, and notes.
8. WHEN a Transaction record is deleted by a User, THE Finance_Service SHALL remove it from all budget calculations and health score inputs within 5 seconds.
9. THE Finance_Service SHALL support pagination for transaction list queries, returning a maximum of 100 records per page with cursor-based navigation.
10. THE Finance_Service SHALL allow filtering of transactions by date range, category, account, amount range, and keyword search on merchant name or notes.

### Requirement 4: Transaction Categorization

**User Story:** As a User, I want my transactions automatically categorized into meaningful expense groups, so that I can understand my spending patterns without manual effort.

#### Acceptance Criteria

1. THE Categorizer SHALL assign each imported Transaction to a category from a predefined hierarchical taxonomy including at minimum: Food (Eating Out, Groceries, Beverages), Transport (Fuel, Public Transport, Cab), Shopping (Clothing, Electronics, General), Entertainment (OTT, Events, Gaming), Health (Pharmacy, Hospital, Fitness), Utilities (Electricity, Internet, Mobile), Housing (Rent, Maintenance), Finance (EMI, Insurance, Investment), Income (Salary, Freelance, Rental, Interest), and Other.
2. WHEN a Transaction is imported, THE Categorizer SHALL assign a category within 3 seconds using rule-based matching on merchant name and description.
3. THE Categorizer SHALL achieve a minimum categorization accuracy of 80% on a held-out test set of Indian merchant names before deployment to production.
4. WHEN a User manually overrides a transaction's category, THE Categorizer SHALL store the correction and use it to improve future categorization for similar merchants for that User.
5. THE Finance_Service SHALL expose an endpoint allowing a User to bulk-recategorize transactions matching a specific merchant name to a chosen category.
6. WHEN the ML_Service produces an updated categorization model, THE Categorizer SHALL apply the new model to newly imported transactions without requiring a Platform restart.
7. THE Finance_Service SHALL compute and store monthly category-wise spending totals for each User, updated within 10 seconds of any transaction change.

### Requirement 5: Budget Management

**User Story:** As a User, I want to set and manage monthly budgets per spending category, so that I can control my expenses and stay on track with my savings goals.

#### Acceptance Criteria

1. THE Finance_Service SHALL allow a User to create a Budget record specifying a category, a monthly spending limit in INR, and an optional period override (custom start and end dates).
2. THE Finance_Service SHALL generate a suggested budget for each spending category based on the User's trailing 3-month average spending, adjusted by the 50-30-20 heuristic (50% needs, 30% wants, 20% savings), and present it to the User before the User sets a manual budget.
3. WHEN a User's spending in a category reaches 80% of the category's budget limit within a calendar month, THE Notification_Service SHALL send an in-app and email alert to the User.
4. WHEN a User's spending in a category exceeds 100% of the category's budget limit within a calendar month, THE Notification_Service SHALL send an in-app and email overspend alert to the User.
5. THE Finance_Service SHALL compute a real-time budget utilization percentage for each active Budget record, updated within 10 seconds of any new transaction.
6. WHEN a User overrides a suggested budget value, THE Finance_Service SHALL store the override and use the User-specified value for all subsequent calculations.
7. THE Finance_Service SHALL provide a monthly budget summary endpoint returning total budgeted amount, total spent, total remaining, and per-category breakdown for a given month.
8. IF a User has no budget set for a category in which a transaction is recorded, THEN THE Finance_Service SHALL flag the transaction as unbudgeted and include it in the monthly summary under an "Unbudgeted" line item.

### Requirement 6: Financial Health Score

**User Story:** As a User, I want a transparent financial health score with explanations, so that I understand my overall financial position and know what to improve.

#### Acceptance Criteria

1. THE ML_Service SHALL compute a Health_Score on a 0–100 scale for each User using a weighted combination of the following sub-scores: savings rate (weight 25%), debt-to-income ratio (weight 25%), expense volatility (weight 15%), emergency fund adequacy in months of expenses (weight 20%), and investment-to-income proportion (weight 15%).
2. WHEN a User's financial data changes (new transaction, updated budget, or updated goal), THE ML_Service SHALL recompute the Health_Score within 60 seconds.
3. THE Finance_Service SHALL store each Health_Score computation as a timestamped snapshot, retaining at least 24 monthly snapshots per User to enable trend visualization.
4. THE Finance_Service SHALL return, alongside each Health_Score, a ranked list of up to 5 improvement actions, each specifying the sub-score it addresses, the recommended action in plain language, and the projected score improvement if the action is taken.
5. WHEN the Health_Score drops by more than 10 points compared to the previous monthly snapshot, THE Notification_Service SHALL send an in-app alert to the User explaining the primary contributing factor.
6. THE Finance_Service SHALL expose a health score trend endpoint returning monthly Health_Score snapshots for the trailing 12 months.
7. WHERE a User has provided fewer than 30 days of transaction data, THE ML_Service SHALL compute the Health_Score using available data and SHALL annotate the score with a low-data-confidence flag visible in the API response.
8. THE ML_Service SHALL store per-feature contribution values for each Health_Score computation so that the Finance_Service can present human-readable explanations of score drivers.

### Requirement 7: Goal Management and Feasibility Analysis

**User Story:** As a User, I want to create financial goals and immediately see whether they are achievable given my current finances, so that I can plan realistically.

#### Acceptance Criteria

1. THE Finance_Service SHALL allow a User to create a Goal record specifying a name, target amount in INR, target date, priority (high, medium, or low), and an optional linked category or investment type.
2. WHEN a Goal is created, THE Finance_Service SHALL compute the required monthly savings amount using the formula: (target amount − current savings allocated to goal) / months remaining, and return it in the creation response.
3. THE Finance_Service SHALL classify each Goal as one of three feasibility statuses: Achievable (required monthly savings ≤ current monthly savings surplus), Stretched (required monthly savings is between 100% and 150% of current surplus), or Unrealistic (required monthly savings exceeds 150% of current surplus).
4. WHEN a Goal is classified as Stretched or Unrealistic, THE Finance_Service SHALL return at least two alternative suggestions: one proposing a reduced monthly contribution with an extended target date, and one proposing a higher monthly contribution with the original target date.
5. THE Finance_Service SHALL recompute Goal feasibility within 60 seconds whenever the User's income, expenses, or savings data changes.
6. THE Finance_Service SHALL support a minimum of 10 simultaneous active Goals per User.
7. WHEN a User marks a Goal as completed, THE Finance_Service SHALL archive the Goal record and exclude it from active feasibility calculations.
8. THE Finance_Service SHALL provide a goals summary endpoint returning all active goals with their current progress percentage, required monthly savings, feasibility status, and days remaining.

### Requirement 8: Subscription Intelligence Tracker

**User Story:** As a User, I want the Platform to automatically detect and track my recurring subscriptions, so that I can identify unused services and avoid surprise renewals.

#### Acceptance Criteria

1. THE Finance_Service SHALL scan a User's transaction history and identify recurring patterns where the same merchant charges a consistent amount (within ±5% variance) at a regular interval (weekly, monthly, quarterly, or annual) across at least 2 consecutive occurrences.
2. WHEN a recurring pattern is detected, THE Finance_Service SHALL create a Subscription record containing merchant name, detected amount, detected frequency, estimated next renewal date, and a confidence score between 0 and 1.
3. THE Finance_Service SHALL present detected Subscriptions to the User for confirmation before marking them as active, and SHALL allow the User to dismiss false positives.
4. THE Finance_Service SHALL maintain a Subscription inventory endpoint returning all active Subscriptions with their monthly equivalent cost, annual equivalent cost, next renewal date, and utilization tag.
5. THE Finance_Service SHALL allow a User to tag each Subscription with one of three utilization levels: Heavily Used, Rarely Used, or Unused.
6. WHEN a Subscription is tagged as Unused or Rarely Used, THE Finance_Service SHALL generate a cancellation or downgrade recommendation showing the projected monthly and annual savings.
7. WHEN a Subscription renewal date is within 7 days, THE Notification_Service SHALL send a reminder notification to the User specifying the merchant name, amount, and renewal date.
8. WHEN a Subscription renewal date is within 3 days and the Subscription is tagged as Unused, THE Notification_Service SHALL send an urgent cancellation reminder to the User.
9. THE Finance_Service SHALL compute a total monthly subscription spend and a total annual subscription spend for the User and include these in the subscription summary endpoint response.
10. THE Finance_Service SHALL detect and flag duplicate Subscriptions where two active Subscription records share the same service category (e.g., two active video streaming services) and present a consolidation recommendation.

### Requirement 9: AI Tax Filing and Optimization System

**User Story:** As a User, I want guided assistance preparing my income tax return with auto-filled data and deduction suggestions, so that I can minimize my tax liability and reduce filing errors.

#### Acceptance Criteria

1. THE Tax_Service SHALL allow a User to create a TaxProfile for a selected FY specifying residential status (resident, non-resident, or resident but not ordinarily resident), tax regime (old or new), and primary income type (salaried, freelance, or mixed).
2. THE Tax_Service SHALL auto-populate TaxProfile fields by reading categorized transactions from the Finance_Service for the selected FY, mapping Income-category transactions to income fields and Finance-category transactions to deduction fields.
3. WHEN a TaxProfile is auto-populated, THE Tax_Service SHALL present a reconciliation summary showing each auto-filled field, its source transaction(s), and a confidence level (high, medium, or low) based on data completeness.
4. THE Tax_Service SHALL model deductions under Sections 80C (up to ₹1,50,000), 80D (up to ₹25,000 for self and family, up to ₹50,000 for senior citizen parents), 80CCD(1B) (up to ₹50,000 for NPS), and standard deduction (₹50,000 for salaried individuals under old regime) as of the applicable FY rules.
5. WHEN a User's TaxProfile shows unused deduction capacity under any supported section, THE Tax_Service SHALL generate a deduction opportunity alert specifying the section, the remaining eligible amount, and eligible investment or expense categories that could utilize it.
6. THE Tax_Service SHALL compute tax liability under both old and new regimes for the User's income profile and return a regime comparison showing tax payable, effective tax rate, and recommended regime with a plain-language explanation.
7. WHEN TDS data is available (via Form 26AS import or manual entry), THE Tax_Service SHALL compute the net tax payable or refund due and flag any mismatch between declared income and TDS records exceeding ₹1,000.
8. THE Tax_Service SHALL validate the TaxProfile against at least the following rules before allowing export: total 80C deductions do not exceed ₹1,50,000; total 80D deductions do not exceed applicable limits; income fields are non-negative; and all mandatory fields for the selected ITR form type are populated.
9. IF a TaxProfile validation check fails, THEN THE Tax_Service SHALL return a structured error response listing each failed validation rule, the field involved, and a plain-language correction instruction.
10. THE Tax_Service SHALL generate an exportable tax summary in JSON format containing all ITR-relevant fields, suitable for manual entry into the Government e-filing portal or upload to a partner ERI platform.
11. THE Tax_Service SHALL display a prominent disclaimer on all tax computation screens stating that computations are based on User-provided data and publicly available rules, and that the Platform is not a substitute for professional tax advice.
12. THE Audit_Service SHALL log every tax computation event with a timestamp, User identifier, FY, and a hash of the input data to support audit trail requirements.
13. WHERE a User's income includes freelance or business income, THE Tax_Service SHALL compute estimated quarterly advance tax liability and present a payment schedule with due dates.

### Requirement 10: Smart CA Finder

**User Story:** As a User, I want to discover and connect with relevant Chartered Accountants, so that I can get professional help for complex tax, audit, or business needs.

#### Acceptance Criteria

1. THE CA_Service SHALL maintain a CAProfile for each registered CA containing: full name, ICAI membership number, office city, pincode, expertise tags (at minimum: Direct Tax, GST, Startup Advisory, Audit, International Tax), languages spoken, remote availability flag, pricing band (low, medium, high), and a verification status.
2. THE CA_Service SHALL provide a search endpoint accepting filters for city, pincode radius (in km), expertise tags (multi-select), remote availability, and pricing band, and SHALL return matching CAProfiles sorted by rating descending.
3. WHEN a CA search query is submitted, THE CA_Service SHALL return results within 500 milliseconds for queries returning up to 50 results.
4. THE CA_Service SHALL compute and store an aggregate rating for each CAProfile based on User-submitted ratings on a 1–5 scale, updating the aggregate within 5 seconds of a new rating submission.
5. THE CA_Service SHALL display a verification badge on CAProfiles where the CA's ICAI membership number has been validated against the ICAI directory or an equivalent verification process.
6. WHEN a User initiates a lead request to a CA, THE CA_Service SHALL require the User to grant explicit consent (creating a ConsentRecord for the CA-sharing purpose) before transmitting any financial summary data to the CA.
7. WHEN consent is granted and a lead request is submitted, THE CA_Service SHALL transmit a case summary containing only the fields the User has explicitly selected for sharing, encrypted in transit using TLS 1.2 or higher.
8. THE CA_Service SHALL allow a CA to accept or decline a lead request within the Platform, and SHALL notify the User of the CA's response via the Notification_Service within 5 minutes of the CA's action.
9. IF a CA declines a lead request, THEN THE CA_Service SHALL present the User with up to 3 alternative CA recommendations matching the same filters.
10. THE CA_Service SHALL allow Users to submit a rating and optional text review for a CA after a lead interaction, and SHALL moderate reviews for prohibited content before publishing.

### Requirement 11: Social Financial Benchmarking System

**User Story:** As a User, I want to see how my financial metrics compare to anonymized peers with similar profiles, so that I can understand whether my savings and spending habits are healthy relative to my cohort.

#### Acceptance Criteria

1. THE Benchmark_Service SHALL define Cohorts along the following axes: income range (bands of ₹2,50,000 annual income), age band (5-year bands), city tier (Tier 1, Tier 2, Tier 3, or Rural), family status (single, married without dependents, married with dependents), and risk profile (conservative, moderate, aggressive).
2. THE Benchmark_Service SHALL compute and store the following aggregates per Cohort on a weekly basis: median savings rate, median EMI-to-income ratio, median expense ratio per top-level category, and median Health_Score.
3. WHEN a User requests a benchmarking summary, THE Benchmark_Service SHALL assign the User to the most specific matching Cohort and return the User's percentile rank for savings rate, EMI-to-income ratio, and Health_Score relative to that Cohort.
4. THE Benchmark_Service SHALL enforce a minimum Cohort size of 50 Users before publishing any aggregate statistics for that Cohort, and SHALL fall back to the next broader Cohort definition if the minimum is not met.
5. THE Benchmark_Service SHALL use only aggregated, de-identified data in all benchmarking computations and SHALL never expose individual User data in any benchmarking response.
6. WHEN a Cohort aggregate is computed, THE Benchmark_Service SHALL apply differential privacy noise with epsilon ≤ 1.0 to all published aggregate values to prevent re-identification of individuals in small cohorts.
7. THE Benchmark_Service SHALL present benchmarking insights in motivational, non-shaming language (e.g., "You save more than 65% of users with a similar income and city" rather than comparative deficit framing).
8. THE Finance_Service SHALL exclude a User's data from all Cohort aggregations if the User has not granted consent for the benchmarking aggregation purpose.

### Requirement 12: Unified Financial Chatbot Assistant

**User Story:** As a User, I want to ask natural-language questions about my finances and trigger platform actions through a conversational interface, so that I can get quick answers and take action without navigating multiple screens.

#### Acceptance Criteria

1. THE Chatbot_Service SHALL recognize user intents across at minimum the following domains: budget status queries, expense analysis queries, tax queries, investment queries, CA discovery, subscription management, and goal planning.
2. WHEN a User submits a chat message, THE Chatbot_Service SHALL return an initial response within 2 seconds for intent recognition and data retrieval queries.
3. THE Chatbot_Service SHALL retrieve live data from Finance_Service, Tax_Service, Investment_Service, and CA_Service to answer data-specific queries (e.g., "How much have I spent on food this month?").
4. THE Chatbot_Service SHALL support action orchestration for at minimum the following actions: create or adjust a budget, create a goal, initiate the tax questionnaire flow, and initiate a CA search.
5. WHEN the Chatbot_Service cannot determine the User's intent with confidence above 0.7, THE Chatbot_Service SHALL ask a clarifying question rather than returning a potentially incorrect response.
6. THE Chatbot_Service SHALL maintain conversation context across a minimum of 10 consecutive turns within a single ChatSession, allowing follow-up questions without re-stating context.
7. WHEN a User asks an investment-related question, THE Chatbot_Service SHALL include a disclaimer stating that responses represent general educational information and not personalized investment advice, and SHALL recommend consulting a SEBI-registered adviser for product-specific guidance.
8. WHEN a User asks a tax-related question, THE Chatbot_Service SHALL distinguish between general tax rules and User-specific computations, and SHALL clearly label User-specific figures as estimates based on available data.
9. THE Chatbot_Service SHALL store each ChatSession with full turn-by-turn logs, intents, and entities in durable storage for a minimum of 90 days to support debugging and quality improvement.
10. THE Chatbot_Service SHALL decline to answer questions outside the financial domain and SHALL respond with a polite out-of-scope message directing the User to relevant platform features.

### Requirement 13: Personalized Investment Advisory System

**User Story:** As a User, I want goal-based investment recommendations aligned with my risk profile and budget, so that I can build wealth systematically without exceeding my financial capacity.

#### Acceptance Criteria

1. THE Investment_Service SHALL administer a risk profiling questionnaire of at least 8 questions covering time horizon, maximum acceptable loss percentage, income stability, prior investing experience, and liquidity needs, and SHALL derive a risk profile of Conservative, Moderate, or Aggressive.
2. WHEN a risk profile questionnaire is completed, THE Investment_Service SHALL store the result as part of the User's profile and use it as the default risk input for all InvestmentPlan computations.
3. THE Investment_Service SHALL generate an InvestmentPlan for each active Goal, specifying a recommended asset category mix (e.g., equity funds, debt funds, ELSS, index funds, FDs, NPS) and a recommended monthly SIP amount, derived from the Goal's target amount, timeline, and the User's risk profile.
4. WHEN generating an InvestmentPlan, THE Investment_Service SHALL verify that the sum of all recommended monthly SIP amounts across all active Goals does not exceed the User's current monthly savings surplus as computed by the Finance_Service, and SHALL flag a budget conflict if it does.
5. THE Investment_Service SHALL present expected return ranges (conservative, base, and optimistic scenarios) for each InvestmentPlan using publicly available historical category-level return data, and SHALL label these as illustrative projections, not guarantees.
6. THE Investment_Service SHALL highlight ELSS and NPS allocations within each InvestmentPlan and show the corresponding 80C and 80CCD deduction benefit in INR, linked to the User's TaxProfile for the current FY.
7. THE Investment_Service SHALL provide model portfolio recommendations at the asset category level only and SHALL not recommend specific mutual fund schemes, stocks, or other individual securities unless operating under or via a SEBI-registered RIA.
8. THE Investment_Service SHALL display a regulatory disclaimer on all investment recommendation screens stating that recommendations are educational model portfolios and not personalized investment advice under the SEBI RIA framework.
9. WHEN a User's risk profile changes, THE Investment_Service SHALL recompute all active InvestmentPlans and notify the User of material changes (defined as a change of more than 10 percentage points in any asset category allocation) via the Notification_Service.
10. THE Investment_Service SHALL allow a User to override the recommended SIP amount for any Goal and SHALL recompute the projected timeline and feasibility accordingly.

### Requirement 14: ML-Based Recommendation Engine

**User Story:** As a User, I want personalized, actionable financial recommendations based on my spending patterns, so that I can make concrete improvements to my financial health.

#### Acceptance Criteria

1. THE ML_Service SHALL generate a ranked list of up to 5 personalized financial recommendations for each User on a weekly basis, each specifying: the improvement lever (e.g., reduce restaurant spending, cancel unused subscription, increase SIP), the projected monthly savings or score improvement, and the confidence level.
2. WHEN generating recommendations, THE ML_Service SHALL use only the requesting User's own data and anonymized Cohort-level patterns, and SHALL not expose individual data from other Users.
3. THE ML_Service SHALL use a rule-based recommendation engine in v1, with rules including: if savings rate is below 20%, recommend reducing the highest variable expense category by 15%; if emergency fund covers fewer than 3 months of expenses, recommend increasing liquid savings; if 80C deduction capacity is unused, recommend ELSS investment.
4. WHEN a User acts on a recommendation (e.g., reduces spending in the suggested category), THE ML_Service SHALL detect the behavioral change within the next weekly computation cycle and update the recommendation list to reflect the improvement.
5. THE Finance_Service SHALL expose a recommendations endpoint returning the current ranked recommendation list with projected impact values and a timestamp of last computation.
6. THE ML_Service SHALL not use sensitive demographic attributes (religion, caste, gender) as features in any recommendation or scoring model.

### Requirement 15: Data Ingestion and Statement Parsing

**User Story:** As a User, I want to import my financial data from bank statements and brokerage reports, so that the Platform has accurate and complete data for analysis.

#### Acceptance Criteria

1. THE Finance_Service SHALL accept bank statement uploads in PDF and CSV formats from at least the following major Indian banks: SBI, HDFC Bank, ICICI Bank, Axis Bank, and Kotak Mahindra Bank, and SHALL parse them into Transaction records.
2. THE Finance_Service SHALL accept brokerage trade reports in CSV and XLS formats for capital gains computation and SHALL map them to Transaction records with a tax-relevant flag.
3. WHEN a statement is parsed, THE Finance_Service SHALL deduplicate transactions by comparing date, amount, and merchant name against existing records for the same Account, and SHALL not create duplicate Transaction records.
4. THE Finance_Service SHALL provide a parsing status endpoint that returns the current status (queued, processing, completed, or failed) and a summary of imported, skipped, and failed records for each upload job.
5. WHEN a statement parsing job fails, THE Finance_Service SHALL retain the original uploaded file for 7 days to allow reprocessing, and SHALL notify the User of the failure with a plain-language description of the issue.
6. THE Finance_Service SHALL process statement parsing jobs asynchronously and SHALL not block the API response beyond returning a job identifier and an estimated completion time.
7. THE Finance_Service SHALL accept Form 26AS uploads in PDF format and SHALL extract TDS entries to pre-populate the User's TaxProfile for the corresponding FY.
8. FOR ALL valid statement files, parsing then exporting then re-importing the resulting transactions SHALL produce an equivalent set of Transaction records (round-trip property).

### Requirement 16: Notification System

**User Story:** As a User, I want timely, relevant notifications about my finances, so that I can take action before problems arise.

#### Acceptance Criteria

1. THE Notification_Service SHALL deliver notifications via at least two channels: in-app (displayed in the Platform UI) and email.
2. THE Notification_Service SHALL deliver in-app notifications within 30 seconds of the triggering event.
3. THE Notification_Service SHALL deliver email notifications within 5 minutes of the triggering event.
4. THE Notification_Service SHALL support the following notification types: budget threshold alert (80% and 100% of limit), Health_Score drop alert, subscription renewal reminder (7-day and 3-day), goal progress update (monthly), tax filing deadline reminder, and deduction opportunity alert.
5. THE Notification_Service SHALL allow a User to configure notification preferences, enabling or disabling each notification type independently for each delivery channel.
6. WHEN a User disables a notification type for a channel, THE Notification_Service SHALL cease delivering that notification type on that channel within 1 hour of the preference change.
7. THE Notification_Service SHALL not send marketing communications to a User who has not granted consent for the marketing communications purpose.
8. THE Notification_Service SHALL maintain a notification delivery log per User retaining the last 90 days of notification events, accessible via an API endpoint.

### Requirement 17: Security and Data Protection

**User Story:** As a User, I want my financial data protected with industry-standard security controls, so that I can trust the Platform with sensitive information.

#### Acceptance Criteria

1. THE Platform SHALL encrypt all data in transit using TLS 1.2 or higher on all API endpoints and web pages.
2. THE Platform SHALL encrypt all sensitive data fields at rest (including transaction amounts, account numbers, tax data, and personal identifiers) using AES-256 or an equivalent industry-standard algorithm.
3. THE Auth_Service SHALL store User passwords as salted hashes using bcrypt with a minimum cost factor of 12 or an equivalent adaptive hashing algorithm.
4. THE Auth_Service SHALL implement device recognition by storing a device fingerprint on first login and flagging logins from unrecognized devices with an email notification to the User.
5. THE Platform SHALL enforce HTTPS-only access and SHALL redirect all HTTP requests to HTTPS with a 301 response.
6. THE Audit_Service SHALL log all data access events for sensitive resources (tax data, financial summaries, CA-shared documents) including the accessor's User identifier, timestamp, resource type, and action.
7. THE Platform SHALL implement rate limiting on all public-facing API endpoints, rejecting requests exceeding 100 requests per minute per IP address with an HTTP 429 response.
8. THE CA_Service SHALL store all documents shared between Users and CAs in encrypted object storage and SHALL generate time-limited signed URLs (expiring within 1 hour) for document access.
9. THE Platform SHALL conduct and pass an OWASP Top 10 security review before each major release.

### Requirement 18: Performance and Scalability

**User Story:** As a User, I want the Platform to respond quickly and reliably even as the user base grows, so that my experience is not degraded by load.

#### Acceptance Criteria

1. THE Platform SHALL achieve a median API response latency of 300 milliseconds or less for all core read operations (transaction list, budget summary, health score, goal list) under a load of 1,000 concurrent users.
2. THE Finance_Service SHALL cache read-heavy aggregate endpoints (monthly budget summary, health score, subscription summary) with a maximum cache staleness of 60 seconds using Redis.
3. THE Platform SHALL process statement parsing, ML scoring, and benchmarking recomputation jobs asynchronously via a background job queue, ensuring these operations do not impact API response latency.
4. THE Platform SHALL support horizontal scaling of stateless API services without requiring shared in-process state.
5. WHEN a background job queue depth exceeds 500 pending jobs, THE Audit_Service SHALL emit an alert to the operations team within 5 minutes.
6. THE Platform SHALL achieve 99.5% uptime measured on a rolling 30-day basis, excluding scheduled maintenance windows communicated at least 24 hours in advance.

### Requirement 19: Observability and Operations

**User Story:** As an Admin, I want comprehensive logging, metrics, and alerting, so that I can monitor system health and respond to incidents quickly.

#### Acceptance Criteria

1. THE Platform SHALL emit structured JSON logs for every API request including: correlation ID, timestamp, service name, endpoint, HTTP method, response status code, and response latency in milliseconds.
2. THE Platform SHALL propagate a correlation ID across all service calls within a single request chain, enabling end-to-end request tracing.
3. THE Platform SHALL expose a metrics endpoint (compatible with Prometheus scraping) reporting: request rate, error rate, p50/p95/p99 latency, job queue depth, and active user session count.
4. THE Platform SHALL trigger an automated alert to the operations team when the error rate on any service exceeds 1% of requests over a 5-minute rolling window.
5. THE Platform SHALL trigger an automated alert when any background job (statement parsing, ML scoring, benchmarking) fails 3 or more times consecutively for the same job.
6. THE Admin dashboard SHALL display real-time system health metrics including active users, API error rate, job queue depth, and last successful ML scoring run timestamp.
7. THE Admin dashboard SHALL allow an Admin to manage CA profiles (approve, suspend, or update verification status) and SHALL log all Admin actions in the Audit_Service.
8. THE Platform SHALL support feature flags for rolling out new ML models and tax-rule updates without requiring a full deployment, configurable via the Admin dashboard.
